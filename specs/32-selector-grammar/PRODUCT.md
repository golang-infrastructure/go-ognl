# Selector Grammar Product Contract

## Summary

This specification makes selector parsing deterministic across every supported container and public lookup API. It preserves established successful paths while defining numeric segments, whitespace, escaping, Unicode, empty keys, and invalid-selector behavior without adding another selector syntax.

## Behavior

1. A selector is an ordered sequence of path segments separated only by unescaped `.` characters, with unescaped `#` characters acting as expansion operators. An empty selector leaves the current value unchanged; leading, trailing, or repeated dots do not create addressable empty segments.
2. A legal non-negative index spelling is ASCII decimal digits with optional leading zeroes, the same digits with one leading `+`, or a leading `-` followed only by one or more zeroes (for example `-0` and `-00`, retained as index zero for compatibility). The numeric value must fit the supported integer range. A map whose key kind is string, including a named string type, always uses the original segment text—including its sign and leading zeroes—as its exact key; a map whose key kind is int, including a named int type, and slices, arrays, and structs interpret a legal spelling as a base-10 key or positional index.
3. After expansion, numeric dispatch is decided independently for each current element. The same segment can therefore resolve a string key in a `map[string]...` element and an index in a slice, array, struct, or `map[int]...` element within one call.
4. Negative non-zero and other non-index text such as `-1`, `-01`, segment text `1.0` (written `1\.0` in a selector), a sign without digits, or an overflowing decimal is not a legal non-negative index. A string-keyed map still treats the original text as an exact key; int-keyed maps, slices, arrays, and structs fail resolution without wrapping, truncating, or panicking. These are valid selector segments and are not selector-syntax errors.
5. At the start of a segment, an unescaped ASCII space, tab, line feed, or carriage return is ignorable leading whitespace and may be repeated. These characters are not path separators and do not create another segment. After the segment has started, those same characters are literal segment content. A key with leading whitespace is expressed by escaping its first whitespace character, for example `\ leading`; vertical tab, form feed, and non-ASCII whitespace are always literal.
6. Backslash is a general escape introducer, not a whitelist-based escape. When followed by any character, it removes its own syntactic role and makes the following character literal, so an otherwise ordinary sequence such as `\q` addresses `q` and an escaped leading whitespace starts a literal segment.
7. Reserved characters have stable literal spellings: `\.` addresses a literal dot, `\#` addresses a literal hash, and `\\` addresses one literal backslash. These escapes work at the beginning, middle, or end of a segment and may be combined in one key.
8. A backslash with no following character is an invalid selector. Consequently, an odd trailing run of backslashes is invalid, while an even trailing run is valid and represents literal backslashes; this condition is reported without panicking and is identified by the stable exported sentinel `ErrInvalidSelector`.
9. `GetE` validates the complete selector before exposing any traversal result. For an invalid selector it returns an error compatible with `errors.Is(err, ErrInvalidSelector)` and an ineffective Result with `Type()==Invalid` and no partial value, regardless of the input value, an earlier missing segment, or an empty expansion.
10. `Get` applies the same fail-closed validation. For an invalid selector it returns an ineffective Result with `Type()==Invalid`, no partial value, and a non-empty `Diagnosis()` containing an error compatible with `errors.Is(diagnosis, ErrInvalidSelector)`.
11. `Result.Get` and `Result.GetE` use exactly the same grammar, container-aware numeric rules, escape handling, and invalid-selector precedence as top-level `Get` and `GetE`. One invalid selector fails the whole Result call without returning matches from earlier expanded elements.
12. Valid Unicode text is matched by its exact UTF-8 encoding. Selector parsing performs no normalization, case folding, whitespace folding, replacement, or trimming, so canonically equivalent but differently encoded keys remain distinct and escaped Unicode remains byte-for-byte intact.
13. The empty string map key is intentionally not expressible in this grammar. An empty selector retains its identity meaning, and repeated dots are ignored rather than interpreted as empty-key lookups.
14. Existing documented and test-covered successful selectors remain compatible, including ordinary dotted paths, named string and int map keys, positional slice/array/struct access, escaped dots, and adjacent expansion such as `users#`. Quotes and brackets gain no syntactic role, and this contract introduces no bracket form, quoted form, parser option, or alternate public lookup API.
15. Selector validity is separate from lookup success. A valid selector that misses a key, uses an out-of-range index, encounters an incompatible container, or overflows an integer index is never reported as `ErrInvalidSelector`. `GetE` preserves the resolution error while returning an Invalid, ineffective Result with no stale value; failed elements do not re-enter top-level or chained expanded results. `Get` preserves its existing Diagnosis rules, and successful lookups retain their existing Type and value semantics.
